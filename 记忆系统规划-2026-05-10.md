# 圆桌智囊团记忆系统规划 2026-05-10

## 1. 当前状况判断

### 1.1 现在还没有真正的记忆系统
目前项目里已经有的，只能算三类“上下文保留”，还不能算完整记忆：

- 话题快照：保存当前任务、席位、讨论状态、导出结果，方便切回某个话题继续。
- 共享事实包：主持 AI 在讨论前先整理一份共享 research brief，给整桌共用。
- 轮次压缩：讨论过程中有 roundNotes 和最终报告，但它们主要服务当前任务，不是长期记忆。

这些能力解决的是：

- 当前任务不要立刻忘掉。
- 当前桌上的人先共享同一份材料。
- 讨论结束后还能回看。

但它们没有解决下面这两件更关键的事：

- 用户长期习惯记忆：越用越熟，记住用户常见目标、偏好、语气、常用人物、常见工作场景。
- 人物长期记忆：固定人物库里的每个人物都能记住自己参与过哪些讨论、形成过哪些稳定立场、曾经和用户在哪些题上达成过什么共识或分歧。

### 1.2 我的判断
你的理解是对的，而且应该明确分成两大方向：

第一类：用户记忆
- 记住用户习惯、偏好、常用领域、常用角色、常用输出风格。
- 目标是“越用越熟”。

第二类：人物记忆
- 不是单纯记住聊天记录，而是让固定人物库中的每个人物拥有自己的长期记忆。
- 目标是“这个人物下次上桌时，不像第一次见面”。

如果只做第一类，不做第二类，这个产品会更像“带长期上下文的聊天工具”。
如果两类都做，才更像“有连续人格和连续关系的圆桌系统”。

## 2. 目标定义

### 2.1 建议做成四层记忆

#### A. 当前任务记忆
- 生命周期：单个话题内部。
- 作用：保存任务定义、澄清问题、共享事实包、轮次结论、关键证据、未决问题。
- 当前已有一部分雏形。

#### B. 用户长期记忆
- 生命周期：跨话题、跨会话。
- 作用：记住用户偏好和长期习惯。
- 例子：
  - 用户偏好更实战的人，而不是空泛学者。
  - 用户经常讨论经营转型、市场改造、产品策略。
  - 用户更喜欢“先做再汇报”的交互方式。

#### C. 人物长期记忆
- 生命周期：跟着固定人物库中的人物走。
- 作用：记住某个固定人物曾经参加过哪些讨论、形成过哪些稳定判断。
- 例子：
  - 某个“运营负责人”过去参与过几次关于线下转型的讨论。
  - 某个“建筑改造顾问”曾经多次强调空间体验和成本约束。
  - 某个“反方质询者”过去在哪些题上提出过有效反对意见。

#### D. 关系记忆
- 生命周期：跨用户、跨人物、跨话题。
- 作用：记住“谁和谁在什么题上曾经站在一起或发生过冲突”。
- 这是最晚做的一层，不建议一开始就上。

### 2.2 第一阶段先做什么
最稳的顺序不是一口气把四层全做完，而是：

1. 当前任务记忆增强
2. 用户长期记忆
3. 人物长期记忆
4. 关系记忆

原因很简单：

- 当前任务记忆最容易见效。
- 用户长期记忆最直接提升体验。
- 人物长期记忆会明显提升“圆桌连续性”。
- 关系记忆复杂度最高，后做更稳。

## 3. 对这个项目最需要的记忆模型

### 3.1 不是“存聊天记录”这么简单
你这个项目不是普通聊天产品，它需要的是“可检索、可归纳、可演化”的记忆。

所以真正需要的不是单一 chat history，而是三种对象：

- preference memory：偏好记忆
- persona memory：人物记忆
- task memory：任务记忆

### 3.2 我建议的内部数据分层

#### 用户记忆
建议拆成：

- user_preferences
- user_domains
- user_style_preferences
- user_role_preferences

示例：
- 更偏企业家、一线操盘手、现实专家
- 不喜欢空泛学者和无关历史人物
- 更希望最终报告按已确认/未确认分开写

#### 人物记忆
建议拆成：

- persona_identity_memory
- persona_experience_memory
- persona_stance_memory
- persona_relationship_memory

示例：
- 这个人物过去参与过哪些类型任务
- 这个人物过去反复强调什么
- 这个人物在哪些题上与用户更容易达成共识
- 这个人物的历史发言中哪些表达可沉淀成稳定风格

#### 任务记忆
建议拆成：

- task_definition_memory
- task_clarification_memory
- task_evidence_memory
- task_round_memory
- task_conclusion_memory

## 4. GitHub 方案调研

## 4.1 候选方案总览

### 方案 A：Mem0
- GitHub: mem0ai/mem0
- 定位：通用 AI memory layer
- 许可：Apache-2.0
- 适合：用户记忆、会话记忆、agent memory、偏好记忆
- 亮点：
  - 自带 user/session/agent 多层记忆思路
  - 有 Python / TypeScript SDK
  - 有 self-hosted 和 cloud 两条路
  - 更像“直接拿来集成”的 memory layer
- 风险：
  - 产品能力比较全，接进来会引入新的后端依赖
  - 人物长期记忆这部分仍要你们自己做 namespace 和策略设计

### 方案 B：Letta
- GitHub: letta-ai/letta
- 定位：stateful agents 平台
- 许可：Apache-2.0
- 适合：如果未来想把整个圆桌系统逐步升级成 stateful agent 平台
- 亮点：
  - 对“有记忆的 agent”支持很强
  - memory_blocks 这种设计很适合人格/用户档案
  - 长期看更适合复杂 agent orchestration
- 风险：
  - 它更像 agent runtime / platform，不只是 memory layer
  - 对当前这个前端原型来说，接入成本偏高
  - 如果你们只是想先补记忆，Letta 可能有点重

### 方案 C：LangMem
- GitHub: langchain-ai/langmem
- 定位：memory management primitives
- 许可：MIT
- 适合：如果后续后端想建立在 LangGraph / LangChain 体系上
- 亮点：
  - 很轻，适合自定义
  - 热路径记忆和后台记忆都有
  - 对“工具式记忆读写”很友好
- 风险：
  - 更像底层积木，不是开箱即用的平台
  - 如果你们不打算用 LangGraph，它的性价比会下降

### 方案 D：Zep
- GitHub: getzep/zep
- 许可：Apache-2.0
- 定位：context engineering / graph-aware retrieval
- 适合：文档、业务数据、事件、关系型上下文很复杂的场景
- 亮点：
  - 很强的 relationship-aware retrieval
  - 图谱化上下文检索能力强
  - 长期适合做“关系记忆”或复杂 context assembly
- 风险：
  - 官方当前重点明显偏 Zep Cloud
  - README 里明确提到 Community Edition 已弃用，旧社区版已转 legacy
  - 对你们当前阶段来说，不是第一优先选择

## 4.2 我的结论

### 最推荐：Mem0
如果目标是“尽快把真正可用的长期记忆接进当前项目”，我最推荐先看 Mem0。

原因：
- 它最接近你们当前需要的“记忆层”，不是整个平台重做。
- 自带 user/session/agent 记忆概念，和你这个项目的结构很接近。
- Apache-2.0，商业开发和后续收费都没问题。
- 有自托管路线，后面不一定要被云服务绑定。

### 第二推荐：LangMem
如果你们后续准备补一个 Python/Node 后端，并且愿意自己掌控记忆结构，LangMem 很合适。

原因：
- 它够轻，适合你们把“用户记忆”和“人物记忆”设计成自己的 schema。
- MIT license 更宽松。
- 适合技术团队自己掌控记忆策略，而不是依赖平台产品逻辑。

### 长期平台候选：Letta
如果以后你们不只是做一个前端原型，而是要把每个角色都演进成 stateful agent，Letta 值得认真评估。

但我不建议把它作为现在第一步。

### 当前不建议优先：Zep
不是说 Zep 不好，而是它更适合更重的上下文工程和关系图谱场景，而且其开源社区版路线已经明显弱化。
现阶段拿它来补这版项目记忆，不是最短路径。

## 5. 推荐落地路线

## 5.1 阶段一：不改前端交互，只补后端记忆层
目标：把“记忆能力”先做成后台服务，不先改界面。

建议：
- 前端仍然保持当前原型交互。
- 新增一个 memory service。
- 先只支持三类读写：
  - user memory
  - persona memory
  - task memory

### 阶段一建议用 Mem0
接法：
- 给每个用户一个 user_id
- 给每个人物一个 persona_id
- 给每个话题一个 topic_id

命名空间建议：
- users/{user_id}/preferences
- users/{user_id}/habits
- personas/{persona_id}/identity
- personas/{persona_id}/history
- topics/{topic_id}/summary
- topics/{topic_id}/rounds

## 5.2 阶段二：把记忆读写接进讨论流
接入点建议：

### 任务整理前
- 先召回用户长期偏好记忆
- 例如：用户喜欢更实战的人，不喜欢空泛学者

### 人物生成前
- 召回用户偏好 + 该任务类型的历史成功角色组合
- 如果人物是固定人物库里的角色，再召回 persona memory

### 角色发言前
- 固定人物先读自己的人物记忆
- 不建议所有角色都读完整聊天历史，要读压缩后的 persona memory block

### 结论报告前
- 召回该任务的任务记忆和历史同类任务结论

## 5.3 阶段三：人物记忆单独做“可成长人格”
这是最有产品特色的一层。

建议每个固定人物至少有以下字段：

- stable_profile：稳定身份设定
- long_term_views：长期形成的判断偏好
- memorable_cases：过去参与过的代表性任务
- relationship_notes：和用户/其他角色的关系备注
- speaking_style_notes：逐步沉淀出的说话风格

这样下次这个人物再上桌时：
- 不只是“再套一次 prompt”
- 而是“带着过去经验回来”

## 5.4 阶段四：关系记忆与案例图谱
只有当上面三层跑稳后，再考虑：

- 哪些人物在哪些任务中经常一起出现
- 哪些角色组合更容易产出高质量结论
- 哪些用户偏好会反复导向某类讨论结构

这一层如果未来真要做强，可以再评估 Zep / Graphiti 这种图谱方案。

## 6. 商业与许可判断

### 6.1 当前最稳的许可选择
- Mem0: Apache-2.0
- Letta: Apache-2.0
- LangMem: MIT
- Zep: Apache-2.0

从 OSS 许可角度看，这几种都可以用于商业开发。

### 6.2 真正要区分的是“用开源自托管”还是“接它的云平台”
需要分开看：

- 开源仓库许可证
- 官方 cloud 服务条款

如果你们后续要收费、要商用、要可控，我的建议是：

- 第一阶段优先自托管或可迁移架构
- 不要一开始把核心记忆绑死在单一 SaaS 平台里

## 7. 我给这个项目的最终建议

### 7.1 结论
你这个项目应该上记忆，而且不是可有可无，是后面产品质感的核心能力。

优先级我建议这样排：

1. 先补用户长期记忆
2. 再补固定人物长期记忆
3. 任务记忆继续强化
4. 最后才做关系记忆

### 7.2 选型建议
如果现在就要定一个最值得先接的方案：

第一选择：Mem0
- 原因：最贴近“给当前项目补记忆层”的需求

第二选择：LangMem
- 原因：如果你们想自己掌控结构，后端工程可控性更高

第三选择：Letta
- 原因：适合以后做 stateful multi-agent 平台化升级

暂不建议优先：Zep
- 原因：现阶段太重，而且当前 OSS 路线不如 Mem0 稳妥直接

## 8. 建议的下一步

1. 先决定是否接受“记忆必须上后端服务”这个前提。
2. 如果接受，优先做一份 memory schema，而不是先写代码。
3. 先用 Mem0 做一个最小可行验证，只接用户长期记忆和任务记忆。
4. 第二步再把固定人物库人物接入 persona memory。
5. 关系记忆和图谱能力放到更后面，不要现在就上。

## 9. 这份规划对当前项目的直接结论

当前项目还没有真正记忆系统。

现在有的是：
- 任务快照
- 共享事实包
- 轮次压缩

还没有的是：
- 用户长期记忆
- 固定人物长期记忆
- 关系记忆

如果按“尽快可落地、后续可商用、不要一开始把项目拖进大重构”的标准来选：

最值得先接的是 Mem0。
