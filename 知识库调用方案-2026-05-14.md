# 通用知识库调用方案（明日实施草案）

## 结论

这个项目可以加一个通用知识库，而且很适合加。

但不建议一上来就做成重量级向量数据库或独立后端检索服务。对当前仓库来说，最合适的第一版是：

`目录清单 + Markdown 文档库 + 分块检索 + 按需注入 prompt`

也就是说：

1. 先准备一个知识库目录清单，让系统先知道“库里有什么”。
2. 模型不会直接读整个知识库，而是先看目录摘要。
3. 需要时，由应用按目录项把对应文档或文档片段取出来，再喂给当前主持人或席位模型。
4. 这样每个模型都可以“随时调用”，但调用动作必须由前端流程控制，不能让模型自己任意读文件。

这个方向是对的，而且比“让所有模型每次都吃完整知识库”更稳、更省 token、更容易维护。

---

## 为什么推荐这一种

当前项目的特点是：

1. 这是一个静态前端原型，核心逻辑集中在 `prototype-ui/app.js`，没有独立后端检索服务。
2. 模型调用发生在浏览器侧，天然更适合本地清单式、轻量式知识注入。
3. 当前已有“共享事实包”“证据链”“角色提示词”这些结构，再加一个可控知识库层，比直接上向量库更顺。

如果一开始就做向量库，会马上遇到这些问题：

1. 需要 embedding 生成链路。
2. 需要文档更新后的重建索引机制。
3. 需要考虑不同模型、不同语言、不同分块策略的兼容性。
4. 需要决定索引存在本地、浏览器、还是额外服务里。

这些都不是不能做，而是对当前项目来说太重了。

所以第一版最稳的方案不是“向量数据库优先”，而是“目录式知识包优先”。

---

## 推荐的第一版结构

建议在根目录新增一个知识库目录，例如：

```text
knowledge-base/
  catalog.json
  docs/
    01-写作约束.md
    02-圣经解释常用框架.md
    03-项目固定世界观.md
    04-证据判断标准.md
    05-常用论证禁忌.md
  chunks/
    01-写作约束.json
    02-圣经解释常用框架.json
```

其中：

### 1. `catalog.json`

这是目录清单。模型不先看全文，而是先看它。

每条建议包含这些字段：

```json
{
  "id": "kb-writing-constraints",
  "title": "写作约束",
  "summary": "规定输出风格、禁忌表达、统一格式要求。",
  "tags": ["写作", "约束", "风格"],
  "whenToUse": "当主持人、裁判或角色需要遵守固定表达规则时",
  "priority": 10,
  "path": "docs/01-写作约束.md"
}
```

### 2. `docs/*.md`

这是人维护的知识正文。

推荐用 Markdown，因为：

1. 你自己好写、好改。
2. 方便后续拆块。
3. 也方便未来导出、版本管理、人工校对。

### 3. `chunks/*.json`

这是给程序检索用的分块结果。

可以手动生成，也可以后续在前端启动时临时切块。第一版为了稳定，建议做成预切块 JSON。

每块建议带：

```json
{
  "docId": "kb-writing-constraints",
  "chunkId": "kb-writing-constraints-03",
  "title": "写作约束",
  "text": "避免统一小标题，不要复述模式要求，不要所有角色都用同一种开头。",
  "tags": ["写作", "角色提示词"],
  "weight": 8
}
```

## 调用方式应该怎么做

你的理解是对的：

知识库需要先有一个“名单目录”，模型才知道里面有没有自己需要的东西。

但这里要注意一个边界：

不是让模型自己随便去翻知识库，而是：

1. 系统先把目录清单喂给当前模型。
2. 模型根据当前任务，判断需要哪几项。
3. 前端再按它选的 `id` 去加载对应知识片段。
4. 把这些片段拼进当前 prompt。

推荐调用链如下：

### A. 主持人开场前

主持人先看目录，判断这轮讨论需要哪些通用规则、方法论或领域背景。

### B. 每个角色发言前

角色先看目录摘要，最多挑 1 到 3 个自己需要的知识项。

### C. 裁判总结前

裁判可以额外调用“证据判断标准”“总结模板约束”“项目固定世界观”等知识项。

---

## 第一版不要做成什么样

第一版不建议这样做：

1. 不要让每个模型每次都读完整知识库。
2. 不要把知识库内容直接混进“证据链”里。
3. 不要先上重量级向量数据库。
4. 不要让模型直接决定文件路径并读取整个文档。

特别是第 2 点要分清：

知识库是“背景规则与常用知识”，不是“本轮证据”。

比如：

1. “证据判断标准”适合进知识库。
2. “某次网页搜索到的一篇文章”适合进证据链。

这两层混在一起，后面很容易把“规则”误当“证据”。

---

## 这种方案和向量库是什么关系

可以这么理解：

### 当前推荐：目录式知识包

适合现在这个项目，优点是：

1. 实现快。
2. 可控性高。
3. 好调试。
4. 易于人工维护。
5. 很适合你这种“我有一些经常要反复调用的通用知识”的场景。

### 后续可升级：向量检索层

等第一版跑顺后，再考虑给 `chunks/*.json` 做 embedding 和相似度召回。

但升级前提是：

1. 目录和文档先稳定。
2. 文档切块策略先稳定。
3. 你已经知道哪些文档常被调用、哪些几乎不用。

否则太早上向量库，只会增加复杂度。

---

## 我建议的第一版检索策略

第一版甚至可以不引入复杂库，直接做一个轻量混合召回：

1. 先按 `catalog.json` 的 `title / summary / tags / whenToUse` 做一次粗匹配。
2. 再对命中的文档块做一次关键词评分。
3. 最后只取前 3 块注入当前 prompt。

如果明天就要落地，我建议优先这样做，而不是先挑库。

如果一定要选一个前端侧轻量搜索库，倾向于：

1. 小规模阶段：`MiniSearch` 或自写简单评分器。
2. 中规模阶段：再考虑更强的浏览器端全文索引。

但就这个项目当前阶段，我更建议先把目录、文档、切块、调用链做对，再决定要不要上库。

## 明天怎么做

明天建议按这个顺序做：

### 第 1 步：建目录结构

新增：

```text
knowledge-base/
  catalog.json
  docs/
  chunks/
```

### 第 2 步：先写 5 到 8 份高频通用知识

优先写你最常复用的那一类，不要一开始铺太多。

比如：

1. 常用写作约束
2. 讨论模式下的输出禁忌
3. 证据强弱判断标准
4. 圣经解释常用框架
5. 项目固定世界观或角色约束
6. 常见逻辑谬误清单
7. 结论报告写法

### 第 3 步：写 `catalog.json`

先把每份文档的标题、摘要、使用场景、标签写好。

### 第 4 步：做切块

先按自然段或小节切，不要一上来做复杂语义切块。

建议每块控制在：

1. 150 到 400 字
2. 一块只表达一个明确规则或一个完整知识点

### 第 5 步：在前端加一个知识库加载器

建议新增几类函数：

1. `loadKnowledgeCatalog()`
2. `searchKnowledgeCatalog(query, role, mode)`
3. `loadKnowledgeChunks(docIds)`
4. `selectKnowledgeForPrompt(context)`

### 第 6 步：接到 prompt 链路里

优先接三处：

1. 主持人开场前
2. 角色发言前
3. 裁判总结前

### 第 7 步：做调试日志

每次调用知识库时，记录：

1. 当前是谁在调用
2. 调用了哪几个文档
3. 注入了哪几个 chunk

这样后面才知道知识库到底有没有真的起作用。

---

## 一个很重要的原则

“每个模型可以随时调用知识库”这句话，正确落地方式不是：

让模型拥有任意读库权限。

而是：

让系统在每次调用前，给模型一份可选目录；模型表达“我需要哪几项”；然后由系统取回并注入。

这样才可控，也更适合当前前端架构。

---

## 明天落地时的推荐目标

明天不要追求“知识库很大”，而要追求“第一条调用链跑通”。

最小可用目标建议是：

1. 能加载目录。
2. 能让主持人或角色从目录里选到相关知识。
3. 能把 1 到 3 个 chunk 稳定注入 prompt。
4. 能在日志里看到本轮到底调了什么。

只要这四件事跑通，后面再扩知识量、再加 embedding、再做更聪明的召回都不晚。
