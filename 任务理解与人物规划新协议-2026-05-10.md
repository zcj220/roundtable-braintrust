# 圆桌智囊团任务理解与人物规划新协议

本文不是设想稿，而是一份可直接接到当前人物生成链路里的执行协议。

目标只有两个：
- 先把任务理解展开，不再只压成三行极简摘要。
- 再让 AI 按任务流程补齐必要人物，而不是直接按职业模板吐人名。

---

## 1. 核心原则

这套协议只写死思考流程，不写死人物答案。

允许写死的内容：
- 先做任务理解，再做人物规划，再做人物实例化。
- 必须检查任务流程、关键现场、关键环节、利益相关方、执行环节、证据解释环节、非常规视角。
- 每一种必要人物类别至少覆盖一次。
- 同类人物允许重复，但不能挤掉关键缺口位。

不允许写死的内容：
- 某类任务固定必须出现哪些职业。
- 用固定行业表去机械套人。
- 因为历史上常见就强行把每桌都塞满名人。

---

## 2. 新链路总览

建议把当前人物生成链路改成三段：

1. 任务理解阶段
2. 人物规划阶段
3. 人物实例化阶段

其中本次先定义前两段。

对应当前代码入口建议：
- requestAiTaskSummary：改为输出“展开版任务理解”。
- requestRoleGenerationIntake：不再直接返回 ready + 简短 planningBrief，而是输出“人物规划结果”。
- requestGeneratedRecommendedRoles：只负责按已确认的任务理解和人物规划结果去实例化人物。

---

## 3. 任务理解阶段的新输出结构

### 3.1 面向用户显示的输出结构

这一步默认不追问，直接先整理一版较完整的任务理解给用户确认。

建议输出 6 段，不要再压成三行：

1. 任务目标
说明这次圆桌最终要解决什么问题，结论要服务什么用途。

2. 任务范围
说明这次重点讨论哪一段，哪些相关内容只是辅助，不要把题目误缩成某个子点。

3. 任务全流程
把这个任务从头到尾拆开，按真实流程写出会经过哪些阶段、哪些节点、哪些关键动作。

4. 关键现场与关键环节
说明哪些地方最容易决定结果，哪些环节最需要不同视角补位。

5. 必要人物方向
先不直接生成人物名，而是说明这件事至少需要哪几类人物视角，每一类为什么需要。

6. 待确认项
最后只给一句确认提示：
“如果这版任务理解有偏差，你直接补充；如果没问题，我就按这个理解继续规划人物。”

### 3.2 推荐展示格式

建议用固定标题块，方便用户补充，也方便后续程序切分：

```text
任务目标：
...

任务范围：
...

任务全流程：
1. ...
2. ...
3. ...

关键现场与关键环节：
- ...
- ...
- ...

必要人物方向：
- ...：为什么需要
- ...：为什么需要
- ...：为什么需要

确认提示：
如果这版理解有偏差，你直接补充；如果没问题，我就按这个理解继续规划人物。
```

### 3.3 供程序内部使用的结构化字段

如果后续要做稳定解析，建议同步约束模型输出一个 JSON 对象，字段如下：

```json
{
  "taskGoal": "",
  "taskScope": "",
  "taskFlow": [
    { "step": 1, "title": "", "detail": "" }
  ],
  "keyScenes": [""],
  "keyDecisionPoints": [""],
  "requiredPerspectiveSeeds": [
    {
      "label": "",
      "reason": "",
      "priority": "high"
    }
  ],
  "optionalPerspectiveSeeds": [
    {
      "label": "",
      "reason": ""
    }
  ],
  "userPreferenceNotes": [""],
  "confirmationPrompt": "如果这版理解有偏差，你直接补充；如果没问题，我就按这个理解继续规划人物。"
}
```

### 3.4 这一阶段必须遵守的规则

- 默认直接整理，不先追问。
- 信息不够时可以保留假设，但要显式写成“当前可先这样理解”。
- 必须展开任务流程，不允许只做一句高度抽象摘要。
- 必须写出“必要人物方向”，但不能直接进入最终人物卡生成。
- 不能因为用户提到一个点，就把整题缩成这个点。

## 4. 任务理解阶段提示词框架

这不是最终代码字符串，而是建议直接改写到 requestAiTaskSummary 的框架。

### 4.1 中文版框架

```text
你是圆桌智囊团里的任务整理 AI。

你的任务不是把用户原话压成三行摘要，而是先整理出一版“可用于配桌”的任务理解稿，等用户确认。

你要像真正懂任务拆解的人一样去理解这件事：
先判断这次讨论最终要解决什么，
再拆出这件事从头到尾会经过哪些阶段，
再指出哪些现场、哪些环节、哪些判断点最关键，
最后总结这件事至少需要哪些人物方向参与。

严格按下面结构输出，使用简体中文，不要寒暄，不要解释过程：

任务目标：
...

任务范围：
...

任务全流程：
1. ...
2. ...
3. ...

关键现场与关键环节：
- ...
- ...
- ...

必要人物方向：
- ...：为什么需要
- ...：为什么需要
- ...：为什么需要

确认提示：
如果这版理解有偏差，你直接补充；如果没问题，我就按这个理解继续规划人物。

要求：
1. 不要只写抽象总结，必须展开任务流程。
2. 不要提前生成最终人物名单。
3. 不要写成行业标签堆砌，要从任务流程去想需要哪些人物方向。
4. 如果信息不够，不要先追问，先基于当前内容给出最佳理解，并把不确定点写进任务范围或关键环节里。
5. 如果这是对已有任务的补充，就保留原任务主轴，只补充缺失条件。

用户需求：{用户输入}
```

### 4.2 这一段的设计重点

这段提示词的重点不是让模型“多写一点”，而是强制它先经过这几个认知动作：
- 目标识别
- 流程拆解
- 关键节点识别
- 人物方向抽取

这样后面人物规划阶段才有稳定输入。

## 5. 人物规划阶段的新输出结构

任务理解被用户确认后，再进入人物规划阶段。

这一步仍然不要直接产出最终人物卡，而是先回答：
这张桌子需要哪些人物类别，哪些是必需位，哪些可以重复，哪些适合借现实人物原型，哪些应从现有人物池复用。

### 5.1 面向程序的输出结构

建议这一步输出一个 JSON 对象：

```json
{
  "planningSummary": "",
  "requiredBuckets": [
    {
      "bucketId": "",
      "label": "",
      "reason": "",
      "mustHave": true,
      "minCount": 1,
      "allowDuplicate": true,
      "preferredSource": "expert"
    }
  ],
  "optionalBuckets": [
    {
      "bucketId": "",
      "label": "",
      "reason": "",
      "preferredSource": "expert"
    }
  ],
  "exemplarCandidates": [
    {
      "bucketId": "",
      "name": "",
      "whyFit": "",
      "modernOverlay": ""
    }
  ],
  "reuseFromLibrary": [
    {
      "name": "",
      "whyReuse": ""
    }
  ],
  "coverageChecklist": [
    {
      "label": "",
      "covered": true,
      "reason": ""
    }
  ],
  "generationRules": {
    "targetCount": 8,
    "maxExemplarCount": 2,
    "avoidDuplicateNames": true,
    "requireHumanReadableNames": true,
    "requireModernKnowledgeOverlay": true
  }
}
```

### 5.2 字段解释

requiredBuckets：
必须覆盖的人物类别。这里不是最终人物名，而是“人物位”。

optionalBuckets：
可补强但不是硬缺口的人物类别。

exemplarCandidates：
如果某个类别适合借现实人物、历史人物、小说人物或高辨识度代表人物，就在这里给候选原型。

reuseFromLibrary：
如果人物库里已有同类人物或同名人物优先复用，放在这里。

coverageChecklist：
给后续人物实例化阶段做兜底检查，避免最后漏位。

generationRules：
集中保存本轮实例化必须遵守的硬约束。

### 5.3 这一阶段必须遵守的规则

- 先规划人物位，不直接输出人物卡。
- 每个必要人物类别至少覆盖一次。
- 同类可以重复，但不能因为重复强势类别而漏掉稀缺关键位。
- 高辨识度人物最多只占总数四分之一。
- 如果现实或历史里有非常贴题的人物原型，可以借其外壳，但必须补上现代知识覆盖层。
- 人物库已有同名同设，禁止重复生成。
- 名称必须说人话，不能是抽象黑话。

## 6. 人物规划阶段提示词框架

这段建议接到 requestRoleGenerationIntake，而不是继续让它直接返回 ready。

### 6.1 中文版框架

```text
你现在不是直接生成人物，而是先为这次圆桌做“人物规划”。

你要先想清楚：
这件事从头到尾会牵涉哪些关键环节，
每个关键环节至少需要什么人来补位，
哪些角色是现场必要角色，
哪些角色是专业判断角色，
哪些角色适合用现实人物、历史人物或高辨识度代表人物来承载。

你的目标不是把职业表列满，而是让整张桌子真正覆盖任务流程。

请严格输出 JSON 对象，不要解释，不要 Markdown。

输出字段必须包含：
- planningSummary
- requiredBuckets
- optionalBuckets
- exemplarCandidates
- reuseFromLibrary
- coverageChecklist
- generationRules

规则：
1. requiredBuckets 里写“必须覆盖的人物类别”，不是最终人物卡。
2. 每个必须类别至少给出一个明确原因。
3. 如果某个类别适合借现实人物或知名角色，在 exemplarCandidates 里给出候选，但不要超过总人数的四分之一。
4. 如果人物库里已有同类人物或同名人物优先复用，写入 reuseFromLibrary。
5. 名称必须人话化，不要输出“边界判断者”“资源整合者”这种抽象标签。
6. generationRules 里要明确：目标人数、最多可用多少 exemplar、是否允许同类重复、是否禁止同名重复、是否要求现代知识覆盖层。
7. 你规划的是“这桌为什么需要这些人物位”，不是直接开始扮演这些人物。

当前已确认的任务理解：
{任务理解}

当前人物池已有名称：
{人物池名称列表}

本轮目标人数：
{目标人数}
```

### 6.2 这一段的设计重点

这一段是整条新链路的关键。

它强制模型在正式生成人物前先经过一次“覆盖率思考”：
- 有没有现场位
- 有没有执行位
- 有没有解释位
- 有没有校正位
- 有没有非常规但有价值的位
- 有没有用户明确偏好的特殊人物

只要这一步稳定了，后面的角色生成就不会再只会吐一串行业模板。

---

## 7. 后续人物实例化阶段必须继承的规则

虽然本次不展开人物实例化 prompt，但为了不跑偏，后续阶段必须继承以下规则：

- 先满足 requiredBuckets，再填 optionalBuckets。
- 如果一个 requiredBucket 能直接复用人物库已有角色，优先复用。
- 若使用 exemplar，必须在 systemPrompt 或 description 里补上“具备现代知识和当代信息”的覆盖层。
- 同名人物在人物库已存在时，禁止重复生成人物。
- 人物名称和 seat 都必须人话化。
- 最终人物数可以 8、10、12，按任务理解和用户要求决定。

## 8. 两个典型例子里的规划方式

### 8.1 复杂事件分析类任务

如果任务是围绕一个复杂事件或争议案件做多视角分析，人物规划不应只落到“刑侦专家 x 3”。

更合理的 requiredBuckets 往往会长成：
- 第一接触或现场进入位
- 现场痕迹解释位
- 医学或鉴定解释位
- 案情推演位
- 制度流程位
- 非常规推理位

这时候 exemplarCandidates 里才适合少量放入高辨识度推理型人物。

### 8.2 产品设计类任务

如果任务是做一个面向市场落地的产品，人物规划不应只落到“设计师 x 3”。

更合理的 requiredBuckets 往往会长成：
- 用户审美位
- 结构与工艺位
- 电子或控制位
- 开模量产位
- 成本供应链位
- 市场与目标用户位
- 场景使用位

这时模型更容易自然想到欧美用户、工业设计、工艺、量产、供应链等真实影响结果的人。

---

## 9. 一句话落地结论

后面真正应该固化进代码的不是职业表，而是这条协议：

先把任务流程想清楚，再把人物位补齐，最后才生成人物。

这才是“让 AI 发散，但不乱飘”的正确做法。
