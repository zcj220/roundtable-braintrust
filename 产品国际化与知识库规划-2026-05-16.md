# 产品国际化与知识库规划（2026-05-16）

## 一、先说结论

这个项目如果准备往海外入口（例如 Product Hunt）上走，**不建议拆成两个独立版本**，也不建议做“中文版网站 + 英文版网站 + 两套逻辑”这种重方案。

当前更合适的是：

1. **一套代码，一套产品结构**
2. **界面支持中英文切换**
3. **默认根据入口场景决定首屏语言**
4. **营销页与应用内文案分层处理**
5. **主题支持浅色 / 深色切换，但深色要改浅一档**
6. **知识库采用“全局知识库 + 项目局部知识包”双层结构**

也就是说：

- 对外发布时，海外入口默认英文
- 国内自己用或中文用户访问时，默认中文
- 进入应用后，用户可以随时切换语言和主题
- 知识库不是每次都全量检索，而是先按目录 / 分类 / 标签缩小范围，再做局部召回

这是成本最低、后续也最好维护的路线。

---

## 二、国际化怎么做最合适

## 2.1 不建议做两个独立版本

不建议：

- 一个中文站
- 一个英文站
- 两套 HTML / 两套文案 / 两套状态逻辑

原因很直接：

1. 你这个项目交互复杂，不只是展示页，而是一个高状态量应用。
2. 讨论流程、人物库、证据链、知识库、模型配置，这些如果拆成两套版本，维护成本会翻倍。
3. 以后每新增一个功能，你都要做两次对齐，极容易出现中英文功能不一致。

所以最稳的方案是：

**单产品、单代码、内建中英切换。**

---

## 2.2 当前项目已经有双语骨架，可以继续扩

现状判断：

1. 当前项目已经有 `appLanguage` 状态。
2. 已经有 `langText()` 和 `UI_TEXT` 这类双语机制。
3. 已经有语言切换入口，不是从零开始。

这意味着：

你不是要“重做国际化”，而是要把现有的**局部双语**推进到**完整产品双语**。

因此不需要重构架构，只需要做这三类补齐：

1. **静态文案补齐**
2. **动态生成文案补齐**
3. **营销与分享场景的英文优先化**

---

## 2.3 面向 Product Hunt 的语言策略

Product Hunt 的默认语境是英文，所以建议：

### 方案：英文优先入口 + 应用内可切换中文

具体做法：

1. **对外展示文案默认英文**
   - 页面标题
   - 简介
   - 功能亮点
   - onboarding 首屏提示
   - 模板人物说明
   - 空态提示

2. **应用内保留中英切换按钮**
   - 顶部工具栏常驻
   - 首次进入可自动判断 `navigator.language`
   - 非中文浏览器默认英文，中文浏览器默认中文

3. **不要做两套部署地址**
   - 不用 `/zh` 和 `/en` 两个站点作为第一阶段必需项
   - 第一阶段只做一个入口
   - 如后续需要 SEO，再考虑独立营销页的语言路由

### 这样做的好处

1. 对 Product Hunt 用户友好
2. 不会破坏你当前中文主工作流
3. 后期想做英文宣传页时也能平滑升级

---

## 2.4 文案层建议拆成两层

### 第一层：营销层文案

面向海外用户，需要更短、更结果导向：

- What it does
- Why not just one model
- How evidence / debate / judge work
- Who it is for

这一层建议英文优先，并单独维护，不要和应用内长文案混在一起。

### 第二层：应用层文案

就是当前应用里的按钮、状态、面板标题、人物库、圆桌台、证据链等。

这一层继续使用当前 `UI_TEXT + langText()` 体系即可。

## 三、主题应该怎么做

## 3.1 不用重做两套 UI，只要把暗色调浅一档

你的判断是对的：

当前项目的深色主题能用，但确实偏黑、偏压。

如果要面向更广用户，尤其是 Product Hunt 这种首次体验场景，建议：

1. 深色主题保留，但整体**提亮 10% 到 18%**
2. 面板不再用接近纯黑的深蓝底
3. 文字和线框对比度维持，但不要“全黑背景 + 高亮字”那种很硬的观感

更合适的方向是：

- 深色背景偏 **烟青 / 石墨蓝 / 暖灰蓝**
- 面板底色偏 **半透明灰蓝**
- 强调色保留暖金 + 冷青
- 背景渐变保留，但对比不要过猛

这类色感更接近你提到的参考项目给人的感觉：

- 有层次
- 不压眼
- 仍然有产品感
- 国际用户也更容易接受

---

## 3.2 主题策略建议

不要做“中英文一个主题，黑白两个版本，各自再单独设计”。

建议做：

1. **浅色主题**：默认对外展示首选
2. **深色主题**：保留给长期使用者
3. **主题状态持久化**
4. **语言与主题独立切换**

推荐默认策略：

- 海外首访：英文 + 浅色
- 中文环境首访：中文 + 深色或跟随系统
- 应用内允许手动切换后记住用户偏好

---

## 3.3 主题改造时优先改哪里

第一批只改变量，不要先大面积重排：

1. 背景主色
2. 面板底色
3. 文字层级色
4. 边框线透明度
5. 按钮高亮色
6. 证据区与讨论区的卡片阴影

也就是说先做 **token 级改造**，不是先重画所有组件。

## 四、知识库应该怎么设计

## 4.1 你的设想是对的：必须分两层

你要的结构，我建议明确成这两层：

### A. 全局知识库

这是长期存在的、跨项目复用的知识层。

适合放：

- 公司介绍
- 品牌定位
- 产品定义
- 固定世界观
- 行业背景
- 写作规则
- 常见 FAQ
- 团队规范
- 业务术语
- 方法论
- 常见角色模板

特点：

- 生命周期长
- 多项目复用
- 不因某次讨论结束而删除

### B. 项目局部知识包

这是某一个项目、某一轮任务、某一段讨论临时带进来的知识。

适合放：

- 本次项目文档
- 本次客户资料
- 本次竞品分析
- 本次附件和网页证据
- 当前轮讨论沉淀出的共享事实
- 只在当前项目有效的约束条件

特点：

- 生命周期短
- 和项目绑定
- 项目结束可删除或归档

### 这两层不能混为一谈

原因是：

1. 全局知识库是“长期背景知识”。
2. 项目局部知识是“当前任务上下文”。
3. 如果混在一起，后面很容易把旧项目资料误带进新项目。

所以你的方向是对的，而且应该明确写进架构里。

---

## 4.2 知识库不应该每次全量检索

这一点也完全正确。

如果每次都扫整个知识库，会有三个问题：

1. 召回噪音高
2. token 成本高
3. 模型会拿到太多弱相关信息，反而影响判断

所以一定要做“先缩范围，再召回”。

---

## 4.3 推荐结构：目录树 + 文档卡 + 分块索引

第一版推荐结构：

```text
knowledge-base/
  catalog.json
  docs/
  chunks/
project-knowledge/
  <project-id>/
    manifest.json
    docs/
    chunks/
```

### `catalog.json`

存全局知识库目录。

每条至少包含：

- id
- titleZh
- titleEn
- category
- tags
- summaryZh
- summaryEn
- audience
- scope
- priority
- sourcePath
- updatedAt

### `docs/`

存原始 Markdown 或文本。

### `chunks/`

存切块后的 JSON，供检索注入。

### `manifest.json`

项目级知识包目录，只记录该项目有哪些临时资料、来自哪里、是否允许讨论时调用。

---

## 4.4 目录应该怎么分，才能方便查找

不要只做一个大搜索框。

建议至少有四种目录维度：

1. **按业务分类**
   - 公司
   - 产品
   - 销售
   - 交付
   - 品牌
   - 法务
   - 技术
   - 方法论

2. **按使用场景**
   - 写作时可调
   - 讨论前预读
   - 裁判总结可调
   - 客户沟通可调
   - 产品设计可调

3. **按作用类型**
   - 背景知识
   - 规则约束
   - 模板示例
   - 事实资料
   - 术语定义

4. **按权限范围**
   - 全局可用
   - 指定项目可用
   - 仅当前轮次可用

这样做的好处是：

不是每次“搜索整个知识库”，而是先限定目录域，再搜局部内容。

---

## 4.5 检索链应该怎么走

推荐一条固定链路：

### 第 1 步：先选作用域

用户或系统先决定：

- 只查项目知识
- 只查全局知识库
- 两者都查

### 第 2 步：按目录过滤

按这些条件先缩范围：

- category
- tags
- role
- mode
- projectId
- useCase

### 第 3 步：再做块级召回

对筛选后的文档块做轻量检索。

### 第 4 步：只注入前 3 到 6 块

而不是把整篇文档送进去。

### 第 5 步：注入时标注来源

例如：

- 来自全局知识库：产品定位 / 品牌语气
- 来自项目知识包：客户 A 需求摘要
- 来自本轮证据链：网页证据 2

这样模型输出时更容易区分“背景知识”和“当前证据”。

---

## 4.6 要不要参考 agency-agents 的知识库

可以参考，但**不能直接照搬**。

它适合参考的点：

1. 有独立知识库页面
2. 有中英双语标题和说明结构
3. 有文档列表 + 当前文档详情的双栏布局
4. 有“选中若干知识条目后注入 Agent 运行”的交互方式
5. 有主题和语言状态统一管理

它不适合直接照搬的点：

1. 它的知识库偏“文档清单”，还不是你这里需要的“全局 + 项目局部 + 证据链协同”结构
2. 它的文档规模和检索深度都偏轻
3. 它是 Agent 工作台式结构，不是圆桌讨论型结构

所以更准确的结论是：

**拿它做 UI 和状态管理参考，不拿它当最终知识库架构。**

---

## 4.7 还可以参考哪类项目

如果要看更成熟的知识库产品形态，可以参考 **Dify** 这一类系统。

它值得参考的点：

1. 把知识库当成独立资源，而不是聊天附件
2. 有 dataset / metadata / chunk / retrieval 配置这些明确层次
3. 检索节点和工作流节点是分开的，说明“知识导入”和“知识调用”不是一回事
4. 支持分块、索引、元数据过滤、召回配置，适合作为二阶段上限参考

但它不适合你现在直接照搬，原因也很明确：

1. 体系太重
2. 需要后端、索引、更多管理界面
3. 会把当前项目从“圆桌讨论产品”拉成“知识平台产品”

所以可以这样理解：

- **agency-agents**：适合参考第一版的 UI、双语、主题、文档挂载方式
- **Dify**：适合参考未来二阶段的知识库深度能力边界

第一版建议靠近前者，结构设计时留出将来向后者升级的空间。

## 五、我建议的产品结构升级

## 5.1 应用内新增两个知识层入口

建议后续在当前项目里出现两个入口：

### 入口 1：全局知识库

用途：

- 管理长期文档
- 建目录
- 建分类
- 维护标签
- 管理是否允许在讨论中被调用

### 入口 2：项目知识包

用途：

- 当前任务临时导入文件
- 挂本次网页证据
- 保存共享事实和项目摘要
- 控制是否参与本次圆桌讨论

这样用户会很清楚：

- 什么是长期资产
- 什么是当前项目的局部上下文

---

## 5.2 圆桌讨论中的调用方式

建议在圆桌台里加一个显式开关：

- 本轮允许调用全局知识库
- 本轮仅调用项目知识包
- 本轮同时允许两者

再加一个精细选项：

- 自动召回
- 手动勾选目录项

这样既能自动，也不会失控。

---

## 5.3 知识库与证据链的边界

必须明确：

### 知识库

是长期可复用的背景知识、规则、模板、组织信息。

### 证据链

是这一次讨论里新增、可追溯、带时间和来源的材料。

### 共享事实包

是主持人把证据链和知识库综合后，压缩出的当前讨论中间层。

这三层都应该保留，但角色不同：

1. 知识库是底库
2. 证据链是现场材料
3. 共享事实包是讨论缓存层

## 七、最终建议（决策版）

如果今天要定方向，我建议直接定成下面这版：

### 国际化

- 一套产品，不拆中英文两个独立版本
- 海外入口默认英文，应用内可切中文
- 先做应用完整双语，再考虑独立营销页 SEO

### 主题

- 保留浅色 / 深色切换
- 深色整体提亮，参考你喜欢的那种烟青暖灰系，不再用过黑底
- 先改变量体系，不先大动结构

### 知识库

- 采用双层知识结构：全局知识库 + 项目局部知识包
- 不与证据链混在一起
- 不做全量检索，先目录过滤，再块级召回
- 第一版用 Markdown + catalog + chunks，暂不上重型向量库

### 参考项目借鉴方式

- 借它的语言切换、主题变量、知识库列表/详情双栏布局
- 不直接搬它的知识库模型
- 你的项目需要更强的“项目级局部知识 + 圆桌讨论调用控制”能力
