# 圆桌智囊团角色系统设计草案

## 1. 先回答核心问题：人物经历重不重要

结论：重要，但不是第一优先级；在第一版产品里不应该把它做成重资产。

### 1.1 为什么说它重要

“人物经历”会影响一个角色怎么看问题、怕什么、信什么、如何表达。

例如：

- 同样是医生，三甲医院急诊医生和社区全科医生，看问题的风险意识就不同
- 同样是牧师，偏讲道型和偏牧养型，对同一段经文的落点也会不同
- 同样是律师，诉讼律师和公司法务，对同一争议的表达重点不同

也就是说，经历会让角色更真实、更稳定、更不像模板。

### 1.2 为什么又说它不是第一优先级

因为第一版产品真正决定输出质量的，不是“经历故事写得多生动”，而是下面这些结构化信息：

1. 这个角色代表谁的利益
2. 这个角色用什么方法分析
3. 这个角色优先相信什么证据
4. 这个角色最关注什么风险
5. 这个角色最后按什么标准下结论

如果这些都没有，单独写一段“他出生在哪里、经历过什么”意义不大。

### 1.3 第一版怎么处理最合适

建议把“人物经历”降级为轻量字段，不做成长篇背景故事，而做成 3 类短字段：

- 职业经历：例如“10 年临床经验”“长期从事青少年牧养”“企业并购法务背景”
- 现实约束：例如“预算敏感”“安全优先”“长期面对一线用户”
- 偏见来源：例如“天然保守”“重视制度”“怀疑权威”“强调个体体验”

这样既保留真实感，又不会把角色系统做得过重。

### 1.4 产品建议

第一版角色卡建议字段优先级如下：

1. 必须有：领域、立场、方法、证据偏好、风险偏好、输出目标
2. 应该有：风格、禁区、常见质疑点
3. 可以有：简短经历
4. 暂时不要重做：完整传记、人生故事、复杂关系网

一句话：

人物经历是加分项，不是底座。第一版可以有，但必须轻量。

## 2. 角色系统的正确做法

不要做“几百个人物库”，要做“角色引擎”。

角色不是靠人工穷举出来的，而是由若干维度动态组合出来的。

建议采用 6 轴模型。

### 2.1 六个核心维度

1. 领域轴

- 医学
- 法律
- 商业
- 技术
- 教育
- 心理
- 神学
- 历史
- 体育
- 公共治理

2. 立场轴

- 用户方
- 专业方
- 机构方
- 公众方
- 反对方
- 中立方
- 执行方
- 审查方

3. 方法轴

- 数据实证
- 历史解释
- 法律解释
- 案例推理
- 成本收益分析
- 伦理审视
- 属灵反思
- 风险控制
- 常识判断

4. 价值轴

- 安全优先
- 效率优先
- 公平优先
- 真实优先
- 合规优先
- 属灵优先
- 商业优先
- 关怀优先

5. 风格轴

- 审慎保守
- 强批判
- 牧养温和
- 法官式中立
- 通俗直白
- 务实执行
- 学术严谨
- 辩论攻击型

6. 经历轴

- 一线实践者
- 长期管理者
- 研究型专家
- 基层使用者
- 受影响群体
- 怀疑者/反方观察者

### 2.2 角色生成逻辑

一个角色不再是“固定人物名”，而是：

角色 = 领域 + 立场 + 方法 + 价值 + 风格 + 轻量经历

例如：

- 运动医学医生 = 医学 + 专业方 + 数据实证 + 安全优先 + 审慎保守 + 一线实践者
- 历史神学家 = 神学/历史 + 专业方 + 历史解释 + 真实优先 + 学术严谨 + 研究型专家
- 无神论历史批评者 = 历史 + 反对方 + 历史解释/证据质疑 + 真实优先 + 强批判 + 怀疑者
- 基层家长 = 教育/公众 + 用户方 + 常识判断 + 关怀优先 + 通俗直白 + 受影响群体

## 3. 整体系统文档草案

## 3.1 系统目标

系统不是要模拟“虚构人格聊天”，而是要帮助用户快速组建一个有分工、有立场差异、能互相质询的讨论席位组合。

系统的目标是：

- 理解用户问题
- 把问题标准化
- 自动推荐 3 到 5 个合适角色
- 允许用户手动调整席位
- 让角色进行结构化讨论
- 输出经受过批判的结论

## 3.2 顶层工作流

1. 用户输入问题
2. 中枢模型理解问题并完成标准化
3. 系统识别问题类型
4. 系统根据规则生成候选角色组合
5. 用户从候选角色中勾选或替换
6. 系统为每个席位分配模型
7. 各角色独立分析
8. 多轮质询与修正
9. 裁判角色形成总结
10. 输出最终结论、分歧点和证据基础

## 3.3 中枢模型需要做的事情

你前面说得对，这里确实应该有一个“大模型中枢”。

这个中枢不负责直接给最终答案，它负责“理解和编排”。

中枢模型职责：

1. 问题标准化

- 把用户的自然表达整理成明确问题
- 提取主题、对象、目标、限制条件

2. 问题分类

- 判断这是事实题、价值题、解释题、决策题还是评审题

3. 角色推荐

- 给出推荐席位
- 说明为什么推荐这些角色

4. 讨论模式推荐

- 推荐独立陈述、交叉质询、投票、裁判总结的组合方式

5. 风险提示

- 提醒用户该问题可能没有唯一答案
- 提醒哪些结论容易受到偏见影响

## 3.4 讨论中的席位设计

不要让用户直接选几十个人物，应该先选席位。

建议第一版固定 5 类席位：

1. 主解释者

- 负责给出正面理解和主要论点

2. 反方挑战者

- 负责拆解漏洞、找反例、提质疑

3. 实务视角者

- 负责把问题放到现实使用场景里看

4. 受影响者/公众视角者

- 负责指出对普通人、用户、群体的真实影响

5. 裁判/综合员

- 负责总结共识、分歧、结论强度和保留意见

用户每次讨论只上 3 到 5 个席位。

## 4. 第一版基础人物原型清单

说明：

- 不建议一开始做 100 个前台可见人物
- 第一版前台建议控制在 18 到 24 个常用原型
- 后台再根据题目动态细化

下面先给出 24 个比较实用的基础原型。

| 编号 | 原型名称 | 主要适用问题 | 核心标签 |
| --- | --- | --- | --- |

| 2 | 批判者 | 需要找漏洞和反例 | 反对方、质疑、反证导向 |
| 3 | 裁判 | 需要归纳结论和争议 | 中立方、总结、裁决 |
| 4 | 普通用户 | 需要通俗性、可用性判断 | 公众方、常识、体验导向 |
| 5 | 企业经营者 | 商业决策、投入产出 | 机构方、商业、效率导向 |

| 10 | 心理咨询师 | 情绪、关系、自我理解 | 心理、关怀、稳态导向 |

| 22 | 运动员/教练 | 训练、执行、身体实践 | 体育、经验、结果导向 |

## 4.1 这 24 个原型如何扩展

每个基础原型都可以细化成子模板。

例如：

医生可细化为：

- 急诊医生
- 临床内科医生
- 运动医学医生
- 精神科医生

法律顾问可细化为：

- 公司法务
- 诉讼律师
- 刑辩律师
- 合规律师

神学家可细化为：

- 历史神学家
- 系统神学家
- 圣经原文研究者
- 护教学者

牧者可细化为：

- 讲道型牧者
- 牧养型长者
- 青年事工同工
- 灵修辅导者

也就是说，前台 24 个原型足够，后台细分模板可以到 80 个以上。

## 5. 角色自动匹配规则表

这一部分是产品里最关键的机制之一。

系统不应该让用户先翻角色列表，而应该先理解题目，再自动给出推荐组合。

## 5.1 第一步：问题分类

建议把问题分成 6 类：

| 问题类型 | 典型例子 | 推荐机制 |
| --- | --- | --- |
| 事实判断题 | “这个说法是真的吗？” | 领域专家 + 批判者 + 裁判 |
| 解释理解题 | “这段经文是什么意思？” | 主解释者 + 历史/文本角色 + 反方 + 裁判 |
| 方案决策题 | “该不该做这个产品？” | 商业/执行/风险/用户 + 裁判 |
| 价值冲突题 | “这样做公平吗？” | 伦理 + 利益相关方 + 反方 + 裁判 |
| 风险评审题 | “这个方案有没有坑？” | 工程/法律/风险/反方 |
| 生活建议题 | “我该怎么处理这段关系？” | 专业角色 + 普通人视角 + 风险角色 |

## 5.2 第二步：识别主题领域

| 主题领域 | 典型关键词 | 优先角色 |
| --- | --- | --- |
| 医学健康 | 症状、治疗、恢复、饮食、心理 | 医生、心理咨询师、普通用户、风险官 |
| 法律合规 | 合同、责任、违法、证据、侵权 | 法律顾问、批判者、裁判 |
| 商业产品 | 用户、市场、转化、成本、增长 | 企业经营者、产品经理、财务视角者、风险官 |
| 技术架构 | 系统、扩展、性能、重构、安全 | 工程师、风险官、产品经理、裁判 |
| 神学经文 | 经文、教义、属灵、教会、解释 | 神学家、牧者、历史学者、无神论批评者、裁判 |
| 教育家庭 | 孩子、学习、家长、老师、成长 | 教师、家长、心理咨询师、普通用户 |
| 公共议题 | 政策、舆论、治理、社会影响 | 记者、执法视角、伦理审查者、普通用户 |
| 体育训练 | 训练、受伤、比赛、表现、恢复 | 运动员/教练、医生、心理咨询师 |

## 5.3 第三步：组合推荐规则

系统最终推荐的不是单个人，而是一个席位组合。

### 规则 A：默认 4 席位结构

适用于大多数问题：

1. 主解释者
2. 反方挑战者
3. 实务/现实视角者
4. 裁判

### 规则 B：争议性高的问题，增加第 5 席

第 5 席通常给：

- 受影响群体代表
- 伦理审查者
- 风险官

### 规则 C：专业问题必须至少有 1 个领域专家

例如医学、法律、神学、工程问题，不能只让普通角色讨论。

### 规则 D：高价值观冲突问题必须至少有 1 个反方角色

否则容易变成一边倒的自我确认。

### 规则 E：输出给普通用户时必须至少有 1 个通俗化角色

否则最后答案可能很强，但不可理解。

## 5.4 自动匹配示例

### 示例 1：圣经章节讨论

用户问题：

“罗马书这段经文到底是在讲得救，还是在讲成圣？”

系统判断：

- 问题类型：解释理解题
- 主题领域：神学经文
- 争议度：高

推荐席位：

1. 神学家
2. 牧者/教会长者
3. 历史学者
4. 无神论批评者
5. 裁判

推荐理由：

- 需要系统解释
- 需要属灵落地
- 需要历史背景
- 需要反方质疑论证强度
- 需要综合员避免偏向单一立场

### 示例 2：产品要不要做

用户问题：

“这个 SaaS 功能现在值不值得做？”

系统判断：

- 问题类型：方案决策题
- 主题领域：商业产品

推荐席位：

1. 产品经理
2. 企业经营者
3. 财务视角者
4. 风险官
5. 裁判

### 示例 3：运动伤病恢复

用户问题：

“膝盖伤后还能不能继续高强度训练？”

系统判断：

- 问题类型：事实判断题 + 生活建议题
- 主题领域：体育训练 + 医学健康

推荐席位：

1. 医生
2. 运动员/教练
3. 心理咨询师
4. 风险官

### 示例 4：法律争议

用户问题：

“这个员工竞业限制协议到底合不合理？”

系统判断：

- 问题类型：风险评审题 + 价值冲突题
- 主题领域：法律合规

推荐席位：

1. 法律顾问
2. 企业经营者
3. 普通用户/员工视角
4. 批判者
5. 裁判

## 6. 人物经历在角色卡里的落地格式

为了避免“人物经历”做得太重，建议每张角色卡只保留这 3 行：

### 角色卡经历模板

- 典型经历：这个角色通常经历过什么
- 现实约束：这个角色最常面对什么压力
- 容易带入的偏见：这个角色可能天然偏向什么

### 示例：牧者/教会长者

- 典型经历：长期带领教会、陪伴信徒、处理现实生命问题
- 现实约束：需要让真理能进入人的处境，而不是停在概念里
- 容易带入的偏见：可能更强调劝勉与安慰，而弱化纯学术争论

### 示例：无神论批评者

- 典型经历：长期从历史、文本、逻辑一致性角度审视宗教主张
- 现实约束：优先要求可验证性和论证一致性
- 容易带入的偏见：可能低估信仰经验和传统解释的内在价值

这已经足够让角色更真实，不需要写成小传。

## 7. 第一版产品建议

如果现在就开始做，建议这样收敛：

### 7.1 第一版先做 18 到 24 个基础原型

不要一开始做 100 个前台角色。

### 7.2 每次只允许上 3 到 5 个席位

不要让用户自由拉满 8 到 10 个角色，否则成本、噪音、重复都会上升。

### 7.3 由中枢模型先推荐，用户再调整

默认推荐比“用户自己空白配置”更重要。

### 7.4 把人物经历做成轻量标签

先不做复杂传记系统。

### 7.5 输出必须结构化

建议固定输出：

- 最终结论
- 共识部分
- 争议部分
- 反方最强质疑
- 回应是否充分
- 结论强度
- 建议下一步行动

---

## 8. 一句话结论

你的方向应该是：

用一个中枢模型先理解问题，再用角色引擎自动推荐几位最合适的讨论者，用户只需要做轻量选择，而不是自己从海量人物库里选人。

“人物经历”有价值，但第一版只需要轻量化地服务于角色稳定性，不应该成为产品复杂度的主要来源。
