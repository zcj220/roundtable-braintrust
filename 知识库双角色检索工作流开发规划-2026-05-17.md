# 知识库双角色检索工作流开发规划

## 1. 本次规划目标

本规划用于收束当前已经确定的产品方向，并作为后续开发执行依据。

当前主目标已经明确：

1. 讨论系统不继续走“每个角色独立从零理解、独立检索、独立回答”的简单聊天流。
2. 改为“主持系统 + 多角色发言 + 发言后交接检索 + 轮次压缩”的状态工作流。
3. 知识库不再只做上传和展示，而要真正进入“本地可整理、可检索、可引用、可被下一位角色消费”的链路。
4. 第一阶段先做最小闭环，不先追求复杂的全自动多轮 Agent 自主规划。

## 2. 前面讨论已确定的设计结论

### 2.1 讨论系统不再是普通聊天框

当前产品本质不是单轮问答，而是：

1. 围绕一个任务进行讨论。
2. 多个角色按顺序发言。
3. 每个角色的职责不同。
4. 中间可能要补本地知识库和网页证据。
5. 用户可能在中途插话、纠偏、补资料。

因此后续系统应按“可暂停、可恢复、可插入检索任务的讨论状态机”来设计，而不是按普通连续聊天来设计。

### 2.2 每个角色一次调用内采用双角色机制

每个角色一次正式运行时，分两个阶段：

#### 阶段 A：发言角色

- 当前角色只以自己的立场发言。
- 不提前替下一个角色发言。
- 不负责决定最终全局结论。

#### 阶段 B：交接角色（handoff）

- 当前角色发言结束后，切换为“下一位资料准备官”。
- 此时不再继续争论，只负责产出下一位角色的资料需求。
- 交接内容只给系统使用，不直接面向用户展示。

这一机制的目的：

1. 把“发言”和“检索规划”拆开。
2. 避免下一位角色从零开始想该查什么。
3. 降低每位角色都要双轮调用的成本。

### 2.3 轮次压缩是必须机制

如果不做轮次压缩，讨论会很快爆 token。

后续上下文分三层：

1. 固定任务层：任务定义、讨论目标、规则、角色列表。
2. 历史压缩层：截至上一轮的轮次摘要，不保留全部历史全文。
3. 本轮现场层：本轮当前发言之前的若干角色完整发言。

原则：

- 全文只在局部保留。
- 摘要才跨轮保留。

## 3. 目标架构

### 3.1 系统名称

建议将后续执行链路命名为：

- 会话状态机 + Handoff 检索工作流

### 3.2 核心状态

第一阶段最少需要以下状态：

1. `topic_ready`：任务已确认，讨论可开始。
2. `speaker_preparing`：当前角色上下文组装中。
3. `speaker_speaking`：当前角色正式发言。
4. `handoff_planning`：当前角色产出给下一位的检索建议。
5. `retrieval_pending`：系统执行本地知识库/网页检索。
6. `retrieval_ready`：检索结果已整理完成。
7. `speaker_resume_ready`：下一位角色资料已备齐。
8. `user_interruption`：用户中途插话、补充、纠偏。
9. `round_complete`：一轮讨论完成。
10. `discussion_complete`：整场讨论结束。

### 3.3 每位角色的运行协议

每位角色一次运行，输出两部分：

#### 第一部分：正式发言

字段建议：

1. `speaker_message`
2. `speaker_claims`
3. `speaker_risks`
4. `speaker_open_questions`

#### 第二部分：handoff 交接

字段建议：

1. `next_role_id`
2. `next_role_focus`
3. `local_knowledge_needed`
4. `web_search_needed`
5. `preferred_categories`
6. `preferred_keywords`
7. `avoid_categories`
8. `missing_evidence_types`
9. `current_round_summary`
10. `recommended_counterpoints`

原则：

- handoff 只写“下一位发言前需要准备什么”。
- handoff 不输出下一位正式发言内容。

### 3.4 下一位角色的输入结构

下一位角色正式发言时，只接收四层输入：

1. 任务定义。
2. 历史压缩摘要。
3. 本轮前面已发言角色的完整发言。
4. 专属准备包：包括 handoff、系统检索结果、必要目录提示。

这一步不再把整套历史全文、整库目录全文全部重发。

## 4. 知识库设计结论

### 4.1 知识库不等于缓存

当前已确定：

1. 模型厂商的 prompt cache 不能替代知识库。
2. 知识库必须是本地可控的检索层。
3. 缓存只解决重复计算和成本问题，不解决资料命中问题。

### 4.2 知识库正确结构

第一阶段知识库至少应包含四层：

1. 原始文件。
2. 标准化文本。
3. 元数据：标题、目录、标签、来源、时间、格式。
4. 检索块（chunk）。

AI 实际消费的是“命中的 chunk + 来源元数据”，不是直接盲读原始文件。

### 4.3 本地知识库第一阶段工作方式

1. 用户上传多种格式资料。
2. 系统先转换为标准文本。
3. 系统生成摘要和 chunk。
4. 系统按目录和标签建立检索入口。
5. 检索时先目录预筛，再做关键词/向量检索。
6. 命中后把片段交给讨论角色使用。

## 5. 当前真正的技术难点

当前真正的难点已经不是“角色如何讨论”，而是：

1. 本地如何接收不同格式资料。
2. 如何统一转成稳定的标准文本。
3. 如何按目录、标签、元数据、chunk 做本地检索。
4. 如何让系统先看目录摘要，再决定是否深检索。
5. 如何把命中结果解释性地展示给用户。

因此后续开发重点应从“讨论形式”转向“本地知识接入与检索基础设施”。

## 6. 参考项目判断

### 6.1 agency-agents 的参考价值

agency-agents 适合参考：

1. 文档管理结构。
2. 手工把选中文档注入 prompt 的方式。
3. 简单知识文档元数据设计。

但它不适合作为本项目最终知识库方案，原因是：

1. 没有完整的自动检索闭环。
2. 没有 chunk 级检索链。
3. 没有真正的本地向量检索与知识门控。
4. 更像“知识文档管理 + 手工注入 prompt”。

### 6.2 ai_syndicate 的参考价值

ai_syndicate 明显更接近本项目真正需要的知识库方向。

从现有规划文档可确认它已经明确采用：

1. 资料导入。
2. 资料分类。
3. 标准化文本转换。
4. 分片文件生成。
5. 本地向量索引与检索。
6. 知识目录摘要与知识门控。
7. 先过滤后检索。

这说明它至少在“本地知识接入、目录治理、向量检索、知识门控”几个关键点上，比当前 roundtable 原型更成熟。

因此当前结论是：

- 不需要先去 GitHub 广泛找更多同类项目。
- 第一优先参考对象应是 ai_syndicate 的后端知识链路。
- 需要重点借鉴的不是 UI，而是后端知识治理思路。

## 7. 第一阶段开发范围

### 7.1 讨论工作流最小闭环

第一阶段只做最小闭环，不一次做复杂自治。

需要完成：

1. 角色发言输出和 handoff 输出的双阶段协议。
2. 轮次压缩摘要。
3. 下一位角色上下文组装。
4. 讨论状态机最少状态。
5. 检索任务回写。

### 7.2 知识库最小闭环

需要完成：

1. 多格式资料上传。
2. 标准文本提取。
3. 元数据整理。
4. chunk 生成。
5. 目录与标签治理。
6. 目录预筛。
7. 本地检索。
8. 命中结果注入。
9. 命中文档与片段展示。

## 8. 推荐实施顺序

### 第一步：定义数据结构

先定义而不是直接写逻辑。

需要先定：

1. `DiscussionState`
2. `SpeakerTurn`
3. `HandoffPlan`
4. `RetrievalTask`
5. `RetrievalResult`
6. `RoundSummary`
7. `KnowledgeDocument`
8. `KnowledgeChunk`

### 第二步：补知识库基础设施

优先级最高。

包括：

1. 上传文件标准化。
2. 文本抽取。
3. 文档分类与目录。
4. chunk 生成。
5. 检索索引。
6. 命中结果结构化输出。

### 第三步：补知识门控

不是每次都查知识库。

要先做：

1. 当前角色是否需要检索。
2. 优先查本地还是网页。
3. 优先查哪些目录。
4. 哪些目录可以跳过。

### 第四步：补双角色运行协议

实现：

1. 角色正式发言。
2. handoff 检索交接。
3. 系统据 handoff 执行检索。
4. 将检索结果组装给下一位角色。

### 第五步：补轮次压缩

实现：

1. 每位角色发言后的微摘要。
2. 每轮结束后的轮次摘要。
3. 下一轮只带摘要，不带旧全文。

### 第六步：补中间状态展示

前端最少展示：

1. 当前角色正在发言。
2. 正在为下一位整理资料。
3. 正在检索本地知识库。
4. 正在补网页证据。
5. 已命中多少条本地/网页证据。
6. 下一位即将开始发言。

## 9. 风险点

### 9.1 双角色污染风险

同一模型在同一次调用里扮演两个角色，可能出现：

1. handoff 内容污染正式发言。
2. 当前角色立场过度影响下一位检索建议。

应对方式：

1. 强约束输出分段。
2. handoff 结构化而不是自由发挥。
3. 系统保留知识门控和目录预筛，不完全相信 handoff。

### 9.2 检索漂移风险

上一位角色给下一位做资料建议，可能方向跑偏。

应对方式：

1. handoff 只是信号之一。
2. 系统还要结合任务定义、角色模板、目录门控一起决定实际检索。

### 9.3 文本标准化风险

不同格式文件解析质量不同，尤其是 PDF、表格、图片。

应对方式：

1. 先做“可用优先”，不要追求第一阶段完美解析。
2. 对解析失败、低质量文本、OCR 不准的情况要做标记。
3. 用户可看到每份资料当前是否已成功标准化。

## 10. 当前结论

当前项目接下来应正式进入开发，而不是继续抽象讨论。

当前优先级顺序已经固定：

1. 先补本地知识库标准化与检索基础设施。
2. 再补讨论状态机与双角色 handoff。
3. 再补轮次压缩。
4. 最后补中间状态展示和用户可见引用。

如果后续还需要参考外部项目，优先继续拆 ai_syndicate 的知识门控、目录摘要、向量索引和本地检索思路，而不是继续泛泛寻找新的“聊天 UI”样板。

## 11. 2026-05-17 补充结论：知识资产必须可见、可下载、可回溯

这部分是本次开发过程中新增确认的产品与工程结论，后续应作为知识库体验的硬要求，而不是可选增强。

### 11.1 只看到“已上传”远远不够

用户不能只看到一条文档记录和一段摘要，就默认系统已经完成知识入库。

真正可用的知识库，至少要让用户能确认以下事实：

1. 原文件是否已经存档。
2. 是否已经完成标准化转换。
3. 标题、正文、元数据是否已经分离整理。
4. 是否已经切成 chunk 并进入检索。
5. 后续讨论到底有没有命中它。

如果这些信息不可见，用户就无法判断系统到底是在“展示文件”，还是在“真正给 AI 建知识资产”。

### 11.2 知识库最少要有三类可见产物

第一阶段不一定先做完整后端任务中心，但至少要把以下三类产物做成用户可见资产：

1. 原文件副本。
2. 标准化 JSON。
3. chunk JSON。

其中：

- 原文件用于审计、回溯和重新处理。
- 标准化 JSON 用于确认“标题、正文、格式、状态、摘要”等字段是否已整理完成。
- chunk JSON 用于确认 AI 最终检索消费的真正对象是什么。

原则：

- AI 最终消费的不是原文件本身，而是“标准化后的结构 + 命中的 chunk”。
- JSON 不是目的，但它是当前阶段最适合作为可下载、可检查、可迁移中间资产的载体。

### 11.3 知识库面板必须显示入库流水

知识库详情页后续至少展示以下信息：

1. 原文件名称与大小。
2. 当前标准化状态。
3. 标准化后的格式类型。
4. chunk 数量。
5. 是否已进入检索。
6. 存档时间。
7. 最近命中记录。

这部分不是“为了好看”的 UI，而是知识治理的基础可观测性。

### 11.4 命中记录必须可回溯

后续系统需要保留最少一层命中记录，至少包括：

1. 命中时间。
2. 触发场景，例如 shared brief、本地检索、下一位角色准备包。
3. 当次检索词。
4. 命中的 chunk 序号。
5. 命中片段摘要。

目的：

1. 让用户知道资料是否真的被 AI 用到了。
2. 让开发阶段可以排查“为什么这次没命中”。
3. 为后续知识门控和检索质量优化留下依据。

### 11.5 当前 roundtable 原型的阶段定位

截至 2026-05-17，本项目中的知识库能力应明确分为两个阶段理解：

#### 阶段 A：当前已落地的原型能力

1. 上传后保留原文件副本。
2. 抽取文本并生成摘要。
3. 生成 chunk。
4. 做前端本地混合检索。
5. 将命中结果注入 shared brief。
6. 在详情中显示最近命中记录。
7. 支持下载原文件、标准化 JSON、chunk JSON。

#### 阶段 B：后续应继续补齐的正式能力

1. 后端独立入库任务。
2. 转换失败重试。
3. OCR 与复杂表格增强。
4. 独立索引目录或向量库存储。
5. 检索日志中心。
6. 版本化重建与重索引。

结论：

- 当前已经不是“只上传文件做展示”的状态。
- 但仍属于“可见、可检查的原型级知识资产链路”，尚未进入完整生产级知识治理。

## 12. 2026-05-17 当前下一段长任务：知识门控驱动的下一位角色准备链

基于今天已经完成的能力，下一段不再继续做零散 UI 修补，而是进入一条更长的主线任务：

- 把“角色发言”正式推进为“角色发言 + handoff 交接 + 知识门控决定 + 下一位准备包”。

这一段长任务的目标不是再多加几个提示词字段，而是把讨论链路真正改造成一个可观察的状态工作流。

### 12.1 当前已完成的前置条件

截至现在，下面这些前置基础已经具备：

1. 本地知识库已经具备原文件、标准化文本、chunk、命中回溯、下载和重建能力。
2. 角色运行时 prompt 已经不再只告诉 AI“你是谁”，而是要求输出正式发言和 handoff 结构。
3. discussion state 已有正式状态对象，并已纳入 topic snapshot。
4. 每位角色发言后，讨论流里已经可以显示一条交接准备消息。

这意味着系统已经不再停留在“普通轮流聊天”的阶段，而是已经有条件进入下一位角色准备链的真正实现。

### 12.2 下一段长任务的核心目标

这一段长任务的最终目标，是让系统在每位角色发言后，能够明确回答下面四个问题：

1. 下一位是谁。
2. 下一位最该看什么。
3. 系统决定先查本地知识还是网页证据。
4. 下一位拿到的准备包到底包含哪些材料。

如果这四个问题没有形成稳定链路，那么 handoff 仍然只是“看起来高级的 JSON”，还不是有效工作流。

### 12.3 分阶段实施

#### 阶段 A：把 handoff 从“返回字段”推进到“可消费准备包”

需要完成：

1. 为 handoff 定义稳定数据结构。
2. 把当前角色的 handoff 挂到 round note 和 live turn 上。
3. 在下一位角色发言前，优先读取上一位 handoff。
4. 将 handoff 中的目录偏好、关键词、证据缺口转成真实检索输入。

验收标准：

- 下一位角色 prompt 中必须能看到上一位 handoff 的有效内容，而不是只有当前轮次上下文。

#### 阶段 B：把知识门控从“静态判断”推进到“检索决策器”

需要完成：

1. 判断当前轮是否需要本地知识检索。
2. 判断当前轮是否需要网页检索。
3. 判断本地知识优先查哪些目录。
4. 判断哪些目录和线索应当跳过。
5. 把检索结果整理成统一 retrieval result，而不是散落在 prompt 里。

验收标准：

- 系统能够在讨论流或状态区清楚显示：当前轮检索决策是什么、命中了哪些资料、哪些证据仍然缺失。

#### 阶段 C：把下一位准备包固定下来

需要完成：

1. 定义 NextSpeakerPackage 数据结构。
2. 包含任务定义、压缩历史、本轮现场、上一位 handoff、知识命中、网页命中、证据缺口。
3. 下一位角色发言时只读取这个准备包，不再临时拼接一堆自由文本。

验收标准：

- 下一位角色的输入来源可解释、可回溯、可打印检查。

#### 阶段 D：把状态展示补成前端可见工作流

需要完成：

1. 在前端显示当前状态：speaker_preparing、speaker_speaking、handoff_planning、retrieval_pending、retrieval_ready、speaker_resume_ready。
2. 显示本轮正在为哪位角色准备资料。
3. 显示检索决策、命中目录、命中数量和仍缺证据。
4. 显示上一位 handoff 与下一位准备包的关键摘要。

验收标准：

- 用户能直接看见“系统当前在为谁准备什么”，而不是只能看到角色轮流发言。
