# 重要备忘

## 2026-05-16 证据链运行方式不要再走错

### 现象
- 之前出现过“证据链不产生”、“证据链乱码”、“后来改成只显示网址”的问题。
- 这次再次排查后确认，不是证据链存储结构本身先坏掉了，核心问题是**本地启动方式走错**。

### 根因
- `prototype-ui/app.js` 的网页搜索证据链，正常情况下会调用本地同源代理接口：`/__roundtable_proxy`。
- 这个代理不是 `python -m http.server` 提供的，而是由 [launcher/serve-static.ps1](launcher/serve-static.ps1) 提供。
- 如果只用 Python 静态服务器打开，例如 `http://127.0.0.1:4173/prototype-ui/`，就可能出现：
	- 网页搜索拿不到结果
	- Wikipedia / DuckDuckGo 请求失败
	- 证据链不生成，或者只出现异常占位

### 正确做法
- 本地调试和给用户看效果时，优先使用：
	- [Start-Dev.bat](Start-Dev.bat)
	- 或直接运行 [launcher/serve-static.ps1](launcher/serve-static.ps1)
- 本次验证可用地址：`http://127.0.0.1:4174/prototype-ui/`

### 正确结果长什么样
- 证据链里会真正出现网页证据条目。
- 条目详情里应该显示**可点击的网址链接**，用户可以自己点开原始网页查看。
- 这次验证成功产出的就是这种形态。截图二里的状态可视为这条链路恢复正常后的参考样子。

### 后续排查顺序
1. 先看当前是不是跑在 `python http.server` 上。
2. 如果是，优先切回 `serve-static.ps1` 再验证，不要先怀疑证据链存储结构。
3. 再检查页面地址是不是代理服务端口，例如 `4174`。
4. 如果证据条目出现并带网址链接，说明证据链主流程是通的。

### 这次相关修复
- 已补一层前端 fallback，避免代理暂时不可用时整条证据链直接失效。
- 但**最稳的方式仍然是使用带代理的本地服务**，不要默认用 Python 静态服务器代替。

### 备注
- 以后如果再出现“上次也是这样说过”的情况，先回看这一条，不要重复从证据存储结构开始误判。

## 2026-05-18 知识库检索方案判断与后续建议

### 当前方案到底够不够
- 以当前原型阶段来看，**现在这套“先按文件分片，再按 chunk 命中”的检索方式是对的，方向没有错**。
- 它至少比“整份长文直接整包喂给 AI”合理很多，因为长文已经先被切成多段，再从命中的那一段里返回摘要片段，不是整篇一起塞进去。
- 但如果资料越来越多、单个文件越来越长、目录越来越杂，**只靠 chunk 命中还不够稳定**，因为它容易出现“片段命中了，但 AI 不知道这份文件整体到底是什么”的问题。

### 我的判断
- 如果站在项目经理、产品经理和多年开发的角度看，我会认为：
- **第一层用目录做范围过滤，第二层用文件级说明做快速判断，第三层再进入 chunk 级命中**，这会比现在更稳。
- 也就是说，目录不是检索内容本身，而是缩小搜索范围；真正应该补的一层，是**每个文件自己的简短总说明**。

### 为什么要给每个文件加一个总说明
- 因为 chunk 命中只能回答“这一段像不像你要找的东西”，但不能很好回答“这整份文件到底讲什么、适不适合继续深挖”。
- 如果一个目录下面有很多长文，只看零散 chunk，AI 很容易：
- 命中局部对，但整体理解偏掉。
- 同时命中多个文件的相似段落，结果混在一起。
- 把同目录下不相关的材料也带进后续判断。

### 我建议的理想顺序
1. 先按目录缩小范围。
2. 再看每个文件的“文件总说明 / 文件摘要 / 文件用途标签”。
3. 确认这份文件大概率对路后，再进入该文件的命中 chunk。
4. 真正喂给 AI 时，优先给“文件说明 + 命中 chunk + 必要时前后 1 段上下文”，而不是整份全文。

### 这个“文件总说明”应该写什么
- 建议控制在几行，够 AI 快速判断即可，不要写成长摘要。
- 最好至少包含：
- 这份文件主要讨论什么。
- 它属于背景资料、规格资料、流程资料、报价资料，还是结论文档。
- 哪些问题适合从它里面找答案。
- 有哪些明显边界，不要把它误当成别的类型资料。

### 当前最务实的结论
- **不是推翻现在的 chunk 检索，而是在 chunk 检索前面补一层“文件说明卡片”**。
- 这是我认为最值得做、性价比最高的一步。
- 这样改完后，AI 会先知道“这份文件大概是什么”，再看命中段，整体会稳很多，也更不容易乱。

### 明天如果继续做，优先级建议
1. 先给每个知识文件补一个简短说明字段，不追求很长。
2. 检索结果里优先显示“文件说明 + 命中 chunk 编号 + 命中片段”。
3. 后续再考虑是否增加“命中 chunk 的前后相邻 chunk 补充上下文”。

### 一句话结论
- 现在这套检索**能用，但还不算稳到可以长期扩资料规模**。
- 我建议保留当前 chunk 检索主干，同时补上“每个文件一条简短总说明”，让 AI 先看说明，再决定要不要深入看这份文件的命中片段。
